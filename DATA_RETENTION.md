# 数据保留、归档与安全清理

所有自动增长数据统一由 `lib/data-retention.ts` 管理。保存路径不再自行静默截断；清理默认只预览，必须显式确认才会修改数据。

## 默认策略

| 数据 | 默认策略 | 保护条件 |
| --- | --- | --- |
| 雷达运行 | 热数据保留 180 天，并始终保留最新 20 条 | 清理前先写入冷归档 |
| 任务运行 | 终态任务保留 30 天 | `queued`、`running` 永不由保留任务删除 |
| 详细学习方案 | 旧版本保留 365 天 | 每个仓库/周期/画像的最新方案，以及存在学习进度的方案 |
| 仓库趋势快照 | 保留 400 天 | 每个仓库的最新一条快照始终保留 |
| 候选仓库 | 180 天未更新后可清理 | 被热雷达、详细方案、反馈或收藏引用时保留 |
| API 限流桶 | 保留 7 天 | 活跃窗口不会被删除 |

环境变量可覆盖上述天数，完整名称见 `.env.example`。下限和上限在代码中强制约束，避免配置成 0 后清空数据。

## 操作方式

先执行预览；该命令不写文件、不归档、不删除数据库行：

```bash
pnpm data:retention
```

核对 JSON 报告中的每类数量后再执行：

```bash
pnpm data:retention -- --apply --confirm=delete-expired-data
```

缺少确认字符串时 apply 会拒绝运行。

## 归档位置

- PostgreSQL：旧雷达完整快照写入 `radar_run_archives`，随后才删除热表快照和派生投影。
- 本地 JSON：旧雷达写入 `.data/archive/radar-runs.json`；可用 `RADAR_RUN_ARCHIVE_FILE` 更改位置。

冷归档不会自动永久删除。正式部署应由维护者把它纳入数据库备份或对象存储生命周期策略。

## 一致性与运行要求

- PostgreSQL 清理在单一事务中执行。
- 本地 JSON 使用临时文件加原子重命名，但多个文件之间无法形成数据库级事务；执行 apply 前应停止本地 Web 和 Worker，完成后再启动。
- 本地模式若中途失败，雷达会先完成归档，因此允许短暂重复，但不会先丢失历史快照。
- 清理不会调用 GitHub 或 DeepSeek。
- `anonymous_sessions` 及其用户数据继续由 Cookie 到期和用户主动清除流程管理，不纳入本轮公共雷达保留命令。
